# ReelFire 产品需求文档（PRD）

| 项目 | 内容 |
|---|---|
| 产品名称 | ReelFire |
| 项目方向 | 方向 B：智能视频精彩片段提取系统 |
| 目标场景 | FPS 玩家个人对战录像精彩片段提取 |
| 重点游戏 | CS:GO / Counter-Strike 2、Valorant |
| 文档版本 | v1.0 |
| 文档状态 | 需求冻结，可进入 MVP 开发 |
| 开发周期 | 1 天 |
| 运行形态 | 本地桌面端网页工作台 |

---

## 1. 产品概述

ReelFire 面向希望回顾、分享或二次剪辑自己游戏表现的 FPS 玩家。玩家上传一段 CS:GO、CS2 或 Valorant 对战录像后，系统使用 OpenCV 按时间采样视频帧，使用自定义 Ultralytics YOLO 模型检测 FPS 画面中的关注目标或事件区域，再结合画面变化和运动强度计算可解释的精彩分数。

系统按分数推荐关键帧和候选片段。玩家可以保留、忽略、排序关键帧，添加标签和备注，并调整片段开始、结束时间。确认后由 FFmpeg 输出适合回看或社交平台发布的粗剪视频，同时保留关键帧联系表和 JSON 分析报告。

产品定位是“机器初选 + 玩家复核”的本地粗剪工具，不承诺完全替代专业视频剪辑，也不对玩家水平作竞技排名。

## 2. 背景与用户问题

FPS 对战录像通常较长，而真正值得分享的内容只占少数时间，例如：

- 连续击杀、残局反杀或一打多；
- 近距离激烈交火；
- 爆破点攻防、拆包或守包；
- 高运动强度的突破、转点或追击；
- 画面变化明显、适合作为短视频开头的镜头。

玩家手工回看存在以下问题：

1. 必须完整观看录像才能定位精彩时刻；
2. 击杀发生前后的铺垫容易被剪掉；
3. 多个相邻高分画面容易形成重复片段；
4. 横屏录像转换为竖屏发布需要额外工具；
5. 关键帧、片段和导出结果缺少统一记录，后续难以继续调整。

## 3. 产品目标

### 3.1 核心目标

1. 降低 FPS 玩家定位精彩时刻所需的人工观看时间；
2. 通过目标密度、画面变化、运动强度给出可解释推荐；
3. 保留人工决定权，允许玩家复核和修改推荐结果；
4. 形成可重新打开的任务记录和标准化交付文件；
5. 在一天内完成可演示、可测试的最小业务闭环。

### 3.2 MVP 成功标准

| 指标 | 验收目标 |
|---|---|
| 任务创建 | 合法视频上传后 2 秒内返回 `job_id`，不等待完整推理 |
| 状态可见 | 页面可观察 `queued -> running -> completed/failed` |
| 真实 CV | 至少一段真实 FPS 视频完成 OpenCV 采样和 YOLO 推理 |
| 推荐结果 | 输出至少 1 张关键帧和 1 个合法候选片段 |
| 可解释性 | 页面或报告展示三项分数及综合分数 |
| 人工复核 | 可以保留/忽略、排序、添加标签和备注并写回 JSON |
| 边界调整 | 可以修改开始/结束时间，后端重新校验合法性 |
| 视频输出 | FFmpeg 成功输出 16:9 和 9:16 两种规格 |
| 可恢复性 | 服务重启后能从历史任务重新打开报告并继续审核 |
| 异常处理 | 格式错误、空文件、模型缺失、分析失败均有明确提示 |

准确率受训练数据、游戏版本、分辨率、HUD 设置和观战视角影响，因此本日验收以“流程真实运行、结果可解释、至少一组素材得到合理候选”为准，不以商业级召回率为硬性条件。

## 4. 目标用户与使用场景

### 4.1 核心用户

| 用户 | 需求 |
|---|---|
| 普通 FPS 玩家 | 快速找到自己录像中的击杀、交火和高强度片段 |
| 游戏内容创作者 | 获得可进一步剪辑的关键帧、片段和竖屏粗剪 |
| 小组演示人员 | 使用固定素材稳定展示完整处理闭环和异常处理 |

### 4.2 典型用户故事

- 作为 FPS 玩家，我希望上传完整对局后自动获得精彩候选，从而不必完整回看视频；
- 作为玩家，我希望看到推荐原因和各项分数，从而判断系统是否可靠；
- 作为内容创作者，我希望保留、忽略并排序关键帧，从而控制最终内容；
- 作为内容创作者，我希望调整片段边界，从而保留击杀前铺垫和击杀后反馈；
- 作为短视频发布者，我希望导出横屏和竖屏版本，从而适配不同平台；
- 作为返回用户，我希望重新打开历史任务，从而继续上次未完成的审核。

## 5. 产品范围

### 5.1 P0：本日必须完成

1. 上传单个 FPS 对战视频；
2. 创建任务并立即返回任务编号；
3. 后台进行 OpenCV 定时采样；
4. 对采样帧执行真实 YOLO 推理；
5. 计算目标、场景变化、运动强度和综合分数；
6. 按分数选择关键帧，并按时间间隔去重；
7. 根据最高分区域生成一个推荐片段；
8. 展示关键帧、检测结果、各项分数和推荐原因；
9. 支持关键帧保留、忽略、排序、标签和备注；
10. 支持调整片段开始和结束时间；
11. FFmpeg 输出 16:9 与 9:16 粗剪视频；
12. 输出关键帧联系表、分析报告和视频；
13. 保存任务状态，支持历史任务重新打开；
14. 完成任务书要求的正常、异常和边界测试。

### 5.2 P1：完成 P0 后开发

- 合并多个候选片段；
- 拖动或按钮调整片段顺序；
- 输出 1:1 方形视频；
- 推荐策略参数在页面可调并保存；
- 生成视频、关键帧和 JSON 报告交付 ZIP；
- 片段标签统计；
- 自动生成封面 Prompt；
- 输出前自动检测原音频是否被保留。

### 5.3 本期明确不做

- 实时游戏画面捕获或直播分析；
- 游戏账号登录、战绩平台或云端同步；
- 反作弊、外挂识别或竞技水平评分；
- 自动发布到抖音、Bilibili、YouTube 等第三方平台；
- 多用户权限、云数据库或分布式任务队列；
- 完整非线性编辑器、字幕编辑器或专业调色；
- 强依赖 OCR 的精确击杀数统计；
- 对所有 FPS 游戏、HUD 皮肤和分辨率作统一准确率承诺。

## 6. 核心业务流程

```text
进入工作台
  -> 输入项目名称并选择 FPS 游戏
  -> 上传对战视频并设置采样间隔、目标时长和输出规格
  -> 系统创建任务并返回 job_id
  -> 页面轮询任务状态
  -> 后台执行视频探测、采样、YOLO 检测和评分
  -> 系统保存关键帧、候选片段和分析报告
  -> 玩家查看关键帧与分数
  -> 玩家保留/忽略、排序、添加标签和备注
  -> 玩家调整片段边界
  -> 系统使用 FFmpeg 输出粗剪视频
  -> 玩家查看或下载报告、联系表和视频
  -> 后续从历史任务重新打开并继续审核
```

## 7. 功能需求

### 7.1 视频上传与任务创建

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-001 | 上传 FPS 对战录像 | P0 | 支持至少 MP4；可选支持 MOV、AVI、MKV |
| FR-002 | 创建任务 | P0 | 返回 `ok: true`、`job_id` 和 `queued` 状态，不阻塞推理 |
| FR-003 | 输入项目设置 | P0 | 至少包含项目名、游戏类型、采样间隔、目标时长、输出规格 |
| FR-004 | 校验输入 | P0 | 不支持格式、空文件、损坏视频均返回可读错误 |
| FR-005 | 保存原始输入 | P0 | 输入保存到 `outputs/<job_id>/input/`，文件名经过安全处理 |

建议首版上传上限为 2 GB，并允许通过配置修改。演示素材建议控制在 1～5 分钟，降低现场等待风险。

### 7.2 任务状态与历史记录

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-010 | 异步处理 | P0 | 上传请求返回后，后台线程继续分析 |
| FR-011 | 状态流转 | P0 | 只允许 `created -> queued -> running -> completed/failed` |
| FR-012 | 状态展示 | P0 | 页面显示任务编号、素材名、状态及创建/开始/结束时间 |
| FR-013 | 失败留档 | P0 | 分析失败仍保存 `job.json` 和可读 `error` |
| FR-014 | 历史任务 | P0 | 列表可查看历史任务，完成任务可重新打开 |
| FR-015 | 安全删除 | P0 | 只能删除合法任务目录，运行中任务不能删除 |

### 7.3 视频分析与精彩评分

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-020 | 视频探测 | P0 | 获取时长、分辨率、FPS 和有无音轨 |
| FR-021 | 定时采样 | P0 | OpenCV 按秒采样并保留准确时间戳 |
| FR-022 | YOLO 检测 | P0 | 每个采样帧保存类别、置信度和边界框 |
| FR-023 | 目标分数 | P0 | 根据配置的 FPS 标签组和目标数量归一化到 `[0,1]` |
| FR-024 | 场景变化分数 | P0 | 相邻采样帧差异归一化到 `[0,1]` |
| FR-025 | 运动分数 | P0 | 帧差或光流强度归一化到 `[0,1]` |
| FR-026 | 综合分数 | P0 | 按固定权重计算并写入报告 |
| FR-027 | 关键帧去重 | P0 | 时间过近的高分帧只保留更高分者 |

### 7.4 推荐与人工审核

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-030 | 推荐关键帧 | P0 | 按综合分数排序并保存关键帧图片 |
| FR-031 | 推荐片段 | P0 | 围绕最高分时刻生成合法片段，默认目标时长 30 秒 |
| FR-032 | 保留或忽略 | P0 | 修改后写回 `analysis_report.json` |
| FR-033 | 标签与备注 | P0 | 支持至少 `multi_kill`、`clutch`、`entry`、`objective`、`other` |
| FR-034 | 排序 | P0 | 修改 `order` 后刷新页面仍能保留 |
| FR-035 | 边界调整 | P0 | 后端验证 `0 <= start < end <= video_duration` |

| 编号 | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-040 | 横屏输出 | P0 | 输出 16:9 MP4，可正常播放 |
| FR-041 | 第二规格 | P0 | 输出 9:16 MP4，可正常播放 |
| FR-042 | 音频兼容 | P0 | 有音频时尽量保留，无音频时不报错 |
| FR-043 | 联系表 | P0 | 将选中关键帧生成一张可查看的联系表 |

## 8. FPS 精彩评分规则

### 8.1 YOLO 标签组

最终使用小组训练或微调的自定义模型。业务层不把模型类别写死，而通过配置映射为以下标签组：

| 标签组 | 示例类别 | 业务含义 | 建议权重 |
|---|---|---|---:|
| `combatant` | player、enemy-like character | 画面中出现交战角色 | 1.0 |
| `combat_event` | muzzle flash、explosion、hit/kill event region | 交火或击杀反馈区域 | 1.2 |
| `objective` | bomb、spike、defuse/plant region | 目标点相关事件 | 0.8 |
| `weapon_item` | dropped weapon、equipment | 武器或道具信息 | 0.5 |

类别是否能够稳定检测必须由训练数据验证。若首版模型只有部分标签，则报告必须记录真实模型类别，不能伪造未检测到的业务事件。

### 8.2 目标分数

```text
weighted_object_count = sum(class_weight * confident_detection_count)
object_score = min(weighted_object_count / object_normalizer, 1.0)
```

建议只统计置信度不低于 `0.35` 的检测。`object_normalizer` 默认取 `5.0`，并允许算法工程师根据验证素材调整。

### 8.3 场景变化和运动强度

- `scene_change_score`：比较当前采样帧与上一采样帧的灰度直方图或归一化像素差；
- `motion_score`：使用连续帧差异或稠密光流的平均强度；
- 两项原始值均应归一化到 `[0,1]`；
- 第一帧没有上一帧时，相关分数置为 `0` 并在代码中显式处理。

### 8.4 综合分数

```text
highlight_score =
    object_score * 0.45
  + scene_change_score * 0.35
  + motion_score * 0.20
```

报告必须保存三项原始归一化分数、权重和最终分数，前端不得只显示“推荐/不推荐”。

### 8.5 默认参数

| 参数 | 默认值 | 可配置范围或说明 |
|---|---:|---|
| 采样间隔 | 0.5 秒 | 建议 `0.25～2.0` 秒 |
| YOLO 置信度 | 0.35 | 建议 `0.20～0.80` |
| 最大关键帧数 | 10 | 至少返回 1 张有效帧 |
| 关键帧最小间隔 | 5 秒 | 避免同一交火镜头重复推荐 |
| 推荐片段目标时长 | 30 秒 | 大于源视频时长则使用完整视频 |
| 片段前置时间 | 12 秒 | 保留交火发生前的铺垫 |
| 片段后置时间 | 18 秒 | 保留击杀或回合反馈 |
| 输出规格 | 16:9 | 同时支持 9:16 |

默认参数应保存到任务报告，实际实现可以根据性能测试调整，但前后端和文档必须同步。

## 9. 片段推荐规则

1. 将采样帧按 `highlight_score` 从高到低排序；
2. 按最小 5 秒时间间隔选择关键帧，时间过近时只保留高分者；
3. MVP 以最高分关键帧为核心，按 12 秒前置和 18 秒后置构造 30 秒候选；
4. 开头越界时整体向后移动，结尾越界时整体向前移动；
5. 当目标时长大于源视频时，片段为完整视频；
6. 最终强制满足 `0 <= start < end <= video_duration`；
7. 关键帧不足时仍生成已有结果，不把任务错误地标记为失败；
8. 用户修改边界后必须由后端重新校验，不能只依赖浏览器。

## 10. 页面需求

### 10.1 首页/上传区

- 项目名称；
- 游戏选择：CS:GO/CS2、Valorant、其他 FPS；
- 视频文件选择；
- 采样间隔、目标时长和输出规格；
- 创建任务按钮；

### 10.2 任务列表

- 任务编号、项目名、素材名；
- 游戏类型、创建时间、状态；
- 查看、删除操作；
- 加载、空列表和请求失败状态。

### 10.3 分析与审核页

- 原视频基本信息和任务状态；
- 处理进度或加载提示；
- 关键帧缩略图、时间戳和 YOLO 框；
- 三项分数、综合分数和推荐原因；
- 保留/忽略、顺序、标签和备注；
- 片段开始/结束时间输入；
- 保存审核与生成粗剪按钮；
- 输出视频播放器、联系表和报告入口；
- 无结果、失败和历史任务重开状态。

## 11. 数据与文件要求

```text
outputs/<job_id>/
├─ input/                  # 原始上传视频
├─ keyframes/              # 关键帧图片
├─ result/                 # 联系表和粗剪视频
├─ job.json                # 状态、时间和错误
└─ analysis_report.json    # 分数、关键帧、片段和审核结果
```

- 所有路径存储为任务目录内的相对路径；
- JSON 使用 UTF-8；
- 更新 JSON 时先写临时文件再原子替换；
- 失败任务也必须保留 `job.json`；
- 上传文件名需要清理，删除接口需要验证路径不能越出 `outputs/`；
- 本地项目不上传玩家录像、输出结果或模型大文件到 Git 仓库。

## 12. API 范围

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` | `/api/health` | 检查服务、模型、GPU 和 FFmpeg |
| `POST` | `/api/jobs` | 上传视频并创建任务 |
| `GET` | `/api/jobs` | 获取历史任务列表 |
| `GET` | `/api/jobs/<job_id>` | 获取任务状态和结果摘要 |
| `DELETE` | `/api/jobs/<job_id>` | 删除非运行任务 |
| `POST` | `/api/jobs/<job_id>/analyze` | 提交或重新提交分析 |
| `PATCH` | `/api/jobs/<job_id>/review` | 保存关键帧审核和片段边界 |
| `POST` | `/api/jobs/<job_id>/rough-cut` | 生成粗剪视频 |
| `GET` | `/api/jobs/<job_id>/report` | 获取完整分析报告 |

统一响应：成功包含 `ok: true`；失败包含 `ok: false` 和可读 `error`，并使用合适 HTTP 状态码。

## 13. 非功能需求

| 类别 | 要求 |
|---|---|
| 可用性 | 上传后立即获得任务号；页面具有加载、空、完成和失败状态 |
| 可恢复性 | 服务重启后从本地 JSON 恢复历史任务 |
| 性能 | 使用 GPU 时优先批量推理；演示素材应在可接受等待时间内完成 |
| 安全 | 限制扩展名和大小；安全文件名；删除路径防穿越 |
| 隐私 | 视频只保存在本地；不上传第三方；交付前清理个人录像 |
| 兼容性 | Windows + Conda 为主；有音频和无音频视频均可处理 |
| 可维护性 | 路径、模型、阈值和权重配置化；模块按 API/CV/视频/前端拆分 |
| 可测试性 | 评分、片段边界、任务状态和 API 错误具有独立测试入口 |

## 14. 风险与应对

| 风险 | 影响 | 应对方案 |
|---|---|---|
| 自定义 YOLO 训练数据不足 | FPS 类别检测不稳定 | 提前准备标注小数据集；联调先用基础模型；报告展示真实类别 |
| 采样间隔过大 | 短暂击杀画面被漏掉 | 默认 0.5 秒；允许降低间隔；演示素材提前验证 |
| 采样间隔过小 | 推理耗时过长 | 设置范围和最大采样帧数；GPU 批量推理 |
| HUD、分辨率差异 | 模型泛化下降 | 训练集覆盖两款游戏和常见 16:9 分辨率 |
| 最高分靠近结尾 | 输出片段意外过短 | 边界整体前移，保持目标时长 |
| 源视频无音频 | FFmpeg 映射失败 | 先用 ffprobe 探测，音轨使用可选映射 |
| 前后端字段不一致 | 联调阻塞 | `docs/API.md` 和报告 Schema 作为唯一契约 |
| 模型或视频文件过大 | Git 仓库膨胀 | `.gitignore` 排除模型和输出，交付时单独打包 |

## 15. 验收与发布门槛

发布演示版本前必须满足：

1. 最小闭环完整运行；
2. 至少一段 CS:GO/CS2 或 Valorant 真实素材产生合理候选；
3. 片段边界单元测试覆盖开头、结尾、过长和正常情况；
4. 有音频和无音频视频均能输出；
5. 5 条正常、5 条异常测试有记录；
6. 至少两个真实 Bug 有复现、原因、修复和回归记录；
7. README 能让新成员完成环境配置和启动；
8. PRD、系统设计、API、测试报告、Bug 记录、截图和演示稿齐全；
9. 仓库中不存在玩家隐私视频、API Key、密码或 Conda 环境文件。

详细逐项验收见 [`ACCEPTANCE_CHECKLIST.md`](ACCEPTANCE_CHECKLIST.md)。
